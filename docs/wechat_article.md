# 我用 6 个 AI Agent 造了一个「PPT 生成器」，把控制权还给人类

> 从一份业务报告到一份咨询级 PPT，中间需要多少步？我们的答案是：6 个 Agent、2 个检查点、和一个「不丢信息」的执念。

---

## 起点：为什么不满意现有的 AI PPT 工具？

市面上的 AI 做 PPT，大致分两种路子：

**第一种，模板填空型。** 选个模板，把内容往里塞。问题在于，它不理解你的内容在说什么——只是把段落贴到固定位置，出来的是排版文件，不是演示文稿。

**第二种，一键生成型。** 把文档丢给 LLM，让它一口气把整个 PPT 写出来。看似省事，但你完全不知道它在中间丢了什么。试过的人都知道，生成的 PPT 往往「泛泛而谈」——原文里那些具体的数据、结论、专有名词，不知道去哪了。

两者的共同问题：**你没法干预。** 生出来什么样就是什么样，不满意只能从头再来。

我们的想法很简单——能不能做一个这样的工具：

1. **不丢信息**：原文有的核心数据和结论，PPT 里必须有
2. **每步可控**：大纲让我看、内容让我改，不满意就退回去重来
3. **图表真实**：数据来自原文表格，不是 LLM 编的
4. **输出专业**：生成的是原生 PPTX，图表是矢量对象，打开就能改

---

## 架构：6 个 Agent，流水线式协作

我们把生成 PPT 的过程拆成了 6 个阶段，每个阶段由一个专门的 Agent 负责：

```
用户上传文档
    ↓
ParseAgent ── 解析文档，提取章节结构、表格数据、嵌入图片
    ↓
AnalyzeAgent ── 分析受众和场景，制定叙事策略
    ↓
PlanAgent ── 用金字塔原则规划大纲（SCQA/SCR/AIDA 等框架）
    ↓
🛑 检查点 1：用户审阅大纲，可编辑后确认
    ↓
ContentAgent ── 并行生成每页内容，含图表数据
    ↓
🛑 检查点 2：用户审阅内容，可逐页编辑
    ↓
HTMLDesignAgent ── 设计视觉布局
    ↓
html2pptx.js + ChartRenderer ── 渲染为原生 PPTX
    ↓
下载 .pptx 文件
```

为什么要拆成 6 步而不是一个大 prompt 搞定？因为**单一 LLM 调用的上限太明显了**：

- 输入 20 页文档 + 「帮我做成 PPT」这种宽泛指令，LLM 的注意力会分散，遗漏关键信息
- 单次输出 token 有限，长文档必然被压缩
- 出了问题你不知道是哪一步出错了，没法精准修复

拆成流水线后，每个 Agent 只做一件事，输入精确、输出可验证。而且每步的结果都保存到数据库，用户可以随时回退到任意阶段重跑。

## 迭代：踩过的坑和怎么爬出来的

这个项目不是一次做对的。开发过程中遇到了不少「理论上行得通、实际跑不通」的问题。

### 坑 1：LLM 的调用方式写错了，但程序不报错

这是最隐蔽的一个 bug。我们的 HTMLDesignAgent 调用 LLM 时，传入的是普通字典 `{"role": "user", "content": "..."}` ，但底层的 LLM 客户端期望的是 `ChatMessage` 对象。

正常来说这应该直接报错对吧？但代码里有一个宽泛的 `try/except Exception`，把错误吞掉了，然后默默走了降级逻辑。**结果是 LLM 模板选择功能从上线第一天起就 100% 失败**，但系统看起来在正常运行——只是输出的 PPT 质量差，我们一直以为是 prompt 的问题。

修复方法：把 `ChatMessage` 的自动转换做进了基类（兼容字典和对象两种输入），同时在 except 里对 `TypeError` / `AttributeError` 这类代码错误做透传，不再静默吞掉。

> **教训**：宽泛的异常捕获是调试的天敌。如果一段代码可能出错，让它早点出错比让它默默失败好得多。

### 坑 2：预览组件崩溃——数据里的 null 值

前端用 React 组件渲染图表和架构图的预览。问题出在数据不完美的时候：`diagram_spec.nodes` 可能是 null，`chart.series.values` 可能是空数组。React 组件拿到 null 直接调用 `.map()` 就崩了。

这不是什么高深的问题，但它的危害是**用户看到空白页，不知道发生了什么**。解决方案很朴素：每个预览组件都加数据存在性检查，数据不完整时显示一个「数据不完整」的占位符，而不是白屏。

### 坑 3：信息越压缩越少——LLM 的「过度概括」倾向

这是最头疼的问题。我们的流程是：

1. ContentAgent 根据原文生成每页的 text_blocks（每条 30-60 字）
2. HTMLDesignAgent 拿到 text_blocks，选择模板，填入槽位

问题出在第 2 步：HTMLDesignAgent 的 LLM 不是搬运原文，而是**重新概括**。一条 50 字的 text_block 进去，出来变成 10 个字。几轮下来，原来「东区收入增速连续三季度领先，建议加大资源投入」变成了「东区增长快」，具体数据全没了。

**临时方案**：在 LLM 的 system prompt 里加「内容保留规则」——明确告诉它你的职责是选布局，不是重写内容，必须原样搬运。这个绷带方案效果「好多了」，但不是根治。

**根治方案（设计中）**：让 LLM 只输出布局选择和字段映射（slot_assignments），比如「第 3 条 text_block 放到 bullets[2]」，具体文本由代码搬运，LLM 完全不接触文本内容。这才是正确的架构。

### 坑 4：模板不够用，架构图画不出来

早期只有 6 种模板：并列要点、双栏对比、数据指标、图表聚焦、引用强调、图标网格。但用户上传的文档里有架构图、时间线、象限分析这些内容，全部退化成了并列要点。

我们加了 4 种纯 CSS 模板：分层架构（architecture_stack）、横向时间线（timeline_horizontal）、象限矩阵（quadrant_matrix）、角色列（role_columns）。关键是全部用 HTML+CSS 的 div 实现，不依赖 SVG，这样 html2pptx.js 可以直接渲染，在 PowerPoint 里生成的是可编辑的原生文本框。

### 坑 5：LLM 选不准模板——加入启发式规则和 layout_hint

即使有 10 种模板，LLM 有时还是会选错。比如一个明明是对比内容的页面，LLM 选了「并列要点」而不是「双栏对比」。

两层解决方案：

**第一层：启发式规则。** 纯代码决策树——检测到 "vs"、"对比"、"相较" 等关键词就选双栏对比模板，检测到大量百分比和金额就选数据指标模板。不依赖 LLM，零延迟。

**第二层：layout_hint 贯通。** 让 PlanAgent 在大纲阶段就为每页指定布局建议（parallel_points / comparison / metrics / chart_focus 等），一路透传到 ContentAgent 和 HTMLDesignAgent。到了模板选择环节，layout_hint 直接短路映射到对应模板，完全绕过 LLM 的判断。

## 技术亮点

### 原生矢量图表

生成的图表不是截图，是 PowerPoint 原生图表对象。打开 PPT 后可以直接点击图表修改数据、调整样式，和手动插入的图表完全一致。实现方式是先用 Playwright 把 HTML 渲染成 PPTX 的布局元素，再用 python-pptx 在预留位置注入原生图表。

### 叙事框架而非章节映射

AI 做 PPT 的常见问题：把文档章节一对一映射成幻灯片。一篇 8 章的文档出来就是 8 页 PPT，没有重点、没有逻辑。

PlanAgent 的做法完全不同：根据用户选择的场景（季度汇报、战略提案、竞标 pitch 等）硬编码映射到叙事框架（SCQA、SCR、AIDA 等），然后让 LLM 按框架组织论点序列，每页的 takeaway_message 必须是含动词的完整句子（"东区收入增速连续三季度领先"），不能是名词短语（"东区收入分析"）。

### per-slide 并行生成

ContentAgent 对每页幻灯片独立调用 LLM，4 页并发。好处是：
- 单页输出 token 少（≤1800），不会截断
- 单页失败不影响其他页面
- 用户可以对单页重跑，附带反馈

### 多模型可配

每个流水线阶段可以配不同的模型。比如用便宜的快速模型做内容填充，用推理能力强的模型做大纲规划。支持所有 OpenAI 兼容接口：SiliconFlow、DeepSeek、通义千问、智谱、Moonshot、Ollama 等。

## 写在最后

这个项目目前还在持续迭代中。核心的 6-Agent 流水线已经跑通，2 个检查点的交互流程也基本完善。后续的计划包括：

- **slot_assignments 映射**：让 LLM 完全不接触文本，只做布局选择，彻底解决内容压缩问题
- **更多模板**：支持更多内容形态（流程图、甘特图等）
- **批量生成**：一次上传多份文档，批量生成多份 PPT
